# Research-Nexus Pro：机器人操作领域的智能科研知识图谱系统

## 🏆 参赛项目书 · 科研赛道

---

## 一、项目一句话

**用74篇顶会论文的数据，构建出机器人操作领域最完整的问题-方法-论文三维知识图谱，让科研人员5分钟看懂一个领域需要5个月才能理清的脉络。**

---

## 二、痛点直击

一个刚进入机器人操作领域的研究生，面对"触觉+视觉+策略学习"这个交叉方向：

| 痛点 | 现实 |
|------|------|
| **论文太多读不完** | 仅触觉操作方向就有70+篇核心论文，跨10年、3个子领域 |
| **问题关系理不清** | "3D Diffusion Policy"和"Tactile-Conditioned Diffusion"到底解决的是同一个问题还是不同问题？ |
| **方法演进看不懂** | PointNet → PointNet++ → PointNet4D → DP3，这条线是怎么串起来的？ |
| **空白点找不到** | 哪些问题还没人解决？哪些方法组合还没有人尝试？ |

**我们做过调研：目前没有任何工具能同时解决这四个问题。**

现有的文献管理工具（Zotero、Mendeley）只管存储，不管分析。现有的综述论文写完就过时。我们需要一个**活的、可交互的、持续更新的知识图谱系统**。

## 三、系统架构

### 3.1 三层数据管线

```
论文PDF → Agent深度阅读 → 结构化提取 → 前端可视化
  74篇       每篇提取6类信息    JSON Schema     7个交互视图
```

**第一层：论文发现与阅读**
- 自动收集领域内核心论文（支持arXiv、会议、期刊）
- Agent读取论文全文，提取摘要、方法、实验、引用
- 已有的52篇深度分析文档作为补充语料

**第二层：结构化信息提取**

每篇论文提取6类信息：

| 信息类型 | 字段示例 | 数量 |
|----------|----------|------|
| **问题** | 名称、层级、年份、状态、价值评分 | 30个 |
| **方法** | 名称、类型、验证状态、目标问题 | 60个 |
| **论文** | 标题、年份、类别、方法论、权威度 | 74篇 |
| **引用关系** | 论文→论文引用链 | 1058条 |
| **问题层级** | 父子关系、深度、分支归属 | 4级 |
| **方法关联** | 方法→问题指向、方法继承 | 60组 |

**第三层：可视化交互**

7个联动视图，每个视图回答一个不同的问题：

| 视图 | 回答的问题 |
|------|-----------|
| **Problem Tree** | 这个领域的问题是怎么分层的？ |
| **Time Evolution** | 每个问题是哪年提出的？解决了吗？ |
| **Method → Problem** | 每个方法解决了哪个问题？ |
| **Method Tree** | 方法之间是什么继承关系？ |
| **Dual Tree Fusion** | 问题树和方法树怎么对应的？ |
| **Paper Timeline** | 论文按类别和时间怎么分布的？ |
| **Citation Network** | 论文之间的引用关系是什么？ |

### 3.2 技术栈

| 层 | 技术 | 选择理由 |
|----|------|---------|
| 数据提取 | Python + pdftotext | 零依赖PDF解析 |
| 数据管线 | Python Pipeline | 自动去重、验证、合并 |
| 前端框架 | React 18 + TypeScript | 类型安全、生态完善 |
| 状态管理 | Zustand | 轻量、适合复杂联动 |
| 可视化 | SVG + Framer Motion | 7种自定义视图，性能可控 |
| 构建工具 | Vite 5 | HMR毫秒级热更新 |
| 部署 | Vercel / GitHub Pages | 零配置部署 |

## 四、核心创新

### 4.1 Agent驱动的论文深度阅读

**不同于传统NLP关键词提取**，我们用Agent做真正的"理解式阅读"：

```python
# 提示词设计核心逻辑
"""
你是一个机器人操作领域的专家。请阅读这篇论文并提取：

1. 论文试图解决的**具体问题**（不是大方向，是具体的技术问题）
2. 论文提出的**方法**（包括完整的技术路线）
3. 这个方法**验证状态**如何？（在真实机器人上验证了吗？）
4. 论文引用了哪些**关键前序工作**？
5. 这篇论文的**核心贡献**是什么？（不是摘要翻译，是你自己的判断）

请关联到已有的问题/方法ID，如果现有图谱中没有，请创建新的。
"""
```

**关键设计**：提示词中会注入当前知识图谱的上下文（已有问题、方法、论文），让Agent在提取时自动建立关联。

### 4.2 四级问题层级

我们不是简单地给论文打标签，而是构建了一个**4级问题分解树**：

```
L0: General Robot Manipulation (根目标)
├── L1: Multi-Modal Perception (子领域)
│   ├── L2: 3D Visual Understanding (具体问题)
│   ├── L2: Tactile Encoding (具体问题)
│   │   ├── L3: Self-supervised Tactile Pretraining (技术方向)
│   │   └── L3: Slip Detection (技术方向)
│   └── L2: Cross-modal Fusion (具体问题)
├── L1: Policy Learning (子领域)
│   ├── L2: Diffusion Policy (具体问题)
│   │   ├── L3: 3D Diffusion Policy (技术方向)
│   │   ├── L3: Flow Matching Policies (技术方向)
│   │   └── L3: Consistency Model Policies (技术方向)
│   └── L2: Imitation Learning (具体问题)
└── L1: Foundation Models for Robotics (子领域)
    ├── L2: Vision-Language-Action (具体问题)
    │   ├── L3: VLA + Tactile (技术方向)
    │   └── L3: VLA + 3D Point Cloud (技术方向)
    └── L2: LLM-based Task Planning (具体问题)
```

**这种层级结构的价值**：一个新手可以在5分钟内理解整个领域的问题框架，而不是迷失在74篇论文的细节中。

### 4.3 六十方法的关系网络

60个方法被组织成：
- **继承链**：PointNet → PointNet++ → PointNet4D
- **跨域融合**：Diffusion Policy + Tactile → Tactile-Conditioned Diffusion
- **目标映射**：每个方法明确指向它解决的问题

### 4.4 交互设计哲学

**"一眼看全局，点击看细节"**

- 左侧导航：7个视图一键切换
- 主画布：可缩放、可拖拽、可搜索
- 右侧卡片：点击任意节点展示详细信息（论文摘要、方法描述、关联论文）
- 底部工具：书签、撤销/重做、明暗模式、导出PNG/JSON、演示模式

## 七、应用场景

### 7.1 新人入门

> 一个硕士生刚进入触觉操作方向，打开Problem Tree，5分钟理解"触觉编码→策略学习→灵巧操作"的问题框架，然后通过Time Evolution看到每个问题的发展阶段，最后通过Citation Network找到最值得精读的10篇论文。

### 7.2 选题决策

> 一个博士生想找一个有价值但竞争不激烈的问题。打开Problem Tree，筛选"unsolved"状态+高valueScore的节点，立刻发现"Tactile-Conditioned Diffusion"和"VLA + Tactile"是两个高价值但只有少量论文的方向。

### 7.3 综述写作

> 一个研究员要写综述论文，用Dual Tree Fusion快速生成"问题-方法"对照表，用Paper Timeline生成按时间线的领域演进图，用Export功能导出PNG和JSON数据。

### 7.4 学术评审

> 一个审稿人需要快速了解投稿论文在领域中的位置，通过Citation Network看到该论文引用了哪些工作、被哪些后续工作引用，通过Method Tree看到该方法在整个方法体系中的位置。

## 八、系统可扩展性

### 8.1 领域可扩展

系统不限于机器人操作。通过更换论文库和调整提取提示词，可以快速适配到：
- 自然语言处理
- 计算机视觉
- 自动驾驶
- 生物信息学
- 任何学术领域

### 8.2 数据持续更新

```
新论文发布 → Agent自动阅读 → 提取信息 → 合并到现有图谱 → 前端实时刷新
```

整个流程可以配置为定时任务，让知识图谱保持最新。

### 8.3 协作模式

- 多人同时标注和审核
- 导出JSON供其他工具消费
- API接口供外部系统查询
